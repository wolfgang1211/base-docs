---
title: Throughput and Limits
description: How Base capacity, gas limits, fee limits, data availability, and endpoint limits constrain transaction throughput.
---

Throughput on Base is constrained by several limits at once. There is no single transactions-per-second value that applies to every workload, because transactions consume different amounts of gas, write different amounts of data, and interact with different RPC and infrastructure paths.

## Capacity model

Base capacity is primarily bounded by:

- **L2 block gas limit:** the total gas that can fit in a sealed L2 block.
- **Flashblock gas budget:** the portion of the next L2 block that can be used by each Flashblock while the block is being built.
- **Per-transaction gas maximum:** the maximum gas limit a single transaction can request before it is rejected by the mempool.
- **L1 data availability cost:** the cost and throughput of posting Base transaction data to Ethereum.
- **RPC and provider limits:** rate limits, method availability, and backend capacity on public or hosted endpoints.

Gas-heavy transactions reduce the number of transactions that fit in a block. Data-heavy transactions can also be constrained by L1 data availability costs even when L2 execution gas is available.

## Gas limits

Exact active gas parameters live in [Transaction Ordering](/base-chain/network-information/transaction-ordering). Use that page for the current full block gas limit, Flashblock gas budget, and per-transaction gas maximum.

This page uses those limits to explain throughput planning. When active parameters change, the practical questions stay the same: how much gas each transaction consumes, how much data it posts to L1, and whether the RPC path can handle the requested method volume.

## Fee limits under load

When demand approaches available capacity, fees become the mechanism that controls inclusion. Base transactions pay both an L2 execution fee and an L1 data fee. The L2 base fee adjusts with demand, and Base also uses a minimum base fee that prevents the L2 base fee from dropping below the configured floor.

Current fee parameters, including minimum base fee and EIP-1559 settings, are documented in [Network Fees](/base-chain/network-information/network-fees). The important throughput implication is that low fees can delay inclusion when demand is high, while realistic priority fees can improve inclusion chances without guaranteeing a specific ordering outcome.

## API and endpoint limits

Public endpoints are rate-limited and are not designed for production traffic. Hosted RPC providers may apply request-per-second limits, compute-unit limits, method restrictions, archive-data limits, or WebSocket subscription limits. Those endpoint limits do not change Base protocol capacity, but they can become the practical bottleneck for apps, wallets, indexers, and monitoring systems.

Production systems should use a hosted [RPC Provider](/base-chain/network-information/rpc-providers) or [run a Base node](/base-chain/node-operators/run-a-base-node), then size traffic for the methods they call most often. Heavy methods such as debug traces, large log ranges, and archive reads usually need separate capacity planning from ordinary reads and transaction submission.

## Practical implications

- Split large operations into transactions that fit under the active per-transaction gas maximum.
- Do not estimate throughput by transaction count alone; estimate by gas, data footprint, and RPC method mix.
- Treat Flashblock preconfirmation as lower-latency feedback, not additional final capacity beyond the sealed L2 block.
- Plan separate capacity for user-facing transaction submission, read-heavy app traffic, indexing, and operational monitoring.
- Use [Transaction Ordering](/base-chain/network-information/transaction-ordering) for the exact Flashblock and per-transaction gas mechanics.
