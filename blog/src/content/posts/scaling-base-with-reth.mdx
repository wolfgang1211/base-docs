As part of our efforts to continue scaling Base, the Base core team has been rigorously evaluating the performance of various execution clients.</p><p>Currently, OP Stack chains primarily use <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/ethereum-optimism/op-geth"><u>Geth</u></a>, as it is the canonical OP Stack execution client. Our long-term vision is for Base to be a multi-execution client chain, where there is diversity of clients for both sequencing and validating the chain. This is critical for any high-availability chain. We have already avoided multiple outages on Base by running Reth in conjunction with Geth [<a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://blog.base.dev/scaling-base-with-reth#h-what-incidents-did-operating-reth-and-geth-concurrently-prevent">1</a>]</p><p>Scaling execution client performance is constrained by various hardware limitations (see <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://www.paradigm.xyz/2024/03/how-to-raise-the-gas-limit-1"><u>this blog series from Paradigm</u></a>). While Geth has been a very stable and performant execution client for Base, we’re beginning to encounter challenges due to the increased usage of the network:</p><ul><li><p><strong>Disk Usage Requirements</strong> - The disk space required to operate a Base Geth archive node is growing ~500Gb/week.</p></li><li><p><strong>Provisioning New Nodes</strong> - Given the large database size, when provisioning a new Geth archive node, it takes ~15 hours to download a snapshot and sync to tip.</p></li><li><p><strong>Long Running Database Compactions</strong> - As the amount of write traffic increases, we’ve encountered long running <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/google/leveldb/blob/main/doc/impl.md#compactions"><u>database compactions</u></a>, resulting in Geth nodes falling behind.</p></li></ul><p>Our immediate concern is the storage requirements for running a Base archive node, which impact the first two bullet points. This is especially relevant given that we recommend <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://en.wikipedia.org/wiki/NVM_Express"><u>NVMe</u></a> storage for operating a node, which has a soft limit of 30TB and a maximum limit of 60TB for many node operators.
