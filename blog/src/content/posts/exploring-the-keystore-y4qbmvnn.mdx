The Keystore </h1></div><div class="relative header-and-anchor"><h2 id="h-the-challenge">The challenge</h2></div><p>Managing smart contract wallets across multiple chains is a growing challenge as the ecosystem becomes increasingly multi-chain. Users often deploy their wallets on several chains—such as Ethereum, Base, or Arbitrum—to take advantage of unique features like lower transaction fees or specific dApps. However, maintaining consistent and secure signer configurations across these networks can quickly become cumbersome and error-prone.</p><p>For example, if a user wants to update their wallet’s signer configuration—such as adding a backup signer or transitioning to a multi-signature setup—this typically requires manually updating signer settings on each chain (if it’s even possible). Although the concept may seem straightforward, developing a practical and functional implementation presents its own set of challenges, primarily due to the current limitations in cross-chain state reading.</p><p></p><hr><div class="relative header-and-anchor"><h1 id="h-2-general-keystore-architecture">2. General Keystore Architecture</h1></div><div class="relative header-and-anchor"><h2 id="h-how-to-make-the-keystore-globally-available-cross-chain">How to make the Keystore globally available cross-chain?</h2></div><p>In a context where wallets are deployed across multiple L2s and alt-L1s, establishing a universally accessible central source of truth is challenging.</p><p>Setting aside alt-L1s (which will need to rely on oracle-like solutions, or opt-out of this completely), there are essentially two approaches—<em>both based on the assumption that L2s can, or will eventually be able to, access L1 state*</em>:</p><ol><li><p>Implement the Keystore directly on L1.</p></li><li><p>Commit only the Keystore state on L1.</p></li></ol><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>both based on the assumption that L2s can, or will eventually be able to, access L1 state</u>*: cross-chain state reading is further explored in the <u>Cross-chain State Reading</u> appendix.</em></p></div></div></div></div><div class="relative header-and-anchor"><h3 id="h-implement-the-keystore-directly-on-l1"><strong>Implement the Keystore directly on L1</strong></h3></div><p>The first approach assumes that the Keystore could take the form of a smart contract, directly deployed on L1, where users would submit transactions to update their signer configuration. Then on the L2s, users would have to provide state proofs of the value of their signer configuration on L1, to be allowed to use their latest signers. In this approach the L1 is used to:</p><ol><li><p>Store the full Keystore state</p></li><li><p>Run the authorization logic for each Keystore signer update</p></li></ol><p>The downside of this approach is that it requires sending a transaction to L1 each time a wallet’s signer configuration is updated, which, at the time of writing, would cost around $5 (assuming a simple and low-cost authorize logic).</p><div class="relative header-and-anchor"><h3 id="h-commit-only-the-keystore-state-on-l1"><strong>Commit only the Keystore state on L1</strong></h3></div><p>The second approach mitigates the downside of the first by avoiding the need to send a transaction to L1 for every wallet signer configuration update. Instead, a new component would be introduced to aggregate and batch Keystore signer updates, posting only a single commitment of the updated Keystore state to L1. Sounds like a rollup, right? In this scenario, the L1 would be responsible for:</p><ol><li><p>Storing commitments to the Keystore state</p></li><li><p>Verifying that each new commitment is valid based on the current state</p></li></ol><p>The new component would need to:</p><ol><li><p>Maintain the full Keystore state</p></li><li><p>Execute the authorize logic for each Keystore signer update</p></li><li><p>Aggregate and batch Keystore signer updates into commitments for posting on L1</p></li></ol><p>It’s assumed that storing the Keystore state and executing the authorization logic on the new component is vastly more cost-effective than performing the same operations on L1.</p><p><br>The downside of the first approach (implementing the Keystore directly on L1) makes it impractical for wallets. <strong>Therefore, the remainder of this article will focus on this second approach.</strong></p><div class="relative header-and-anchor"><h2 id="h-how-to-commit-to-the-keystore-state-on-l1">How to commit to the Keystore state on L1?</h2></div><p>In the previous section, we explored how <u>committing the Keystore state to L1</u> enabled efficient cross-chain state sharing (or at least our best attempt at it). However, this approach introduced the need for a new component responsible for processing and applying Keystore signer updates and periodically posting commitments to the Keystore state on L1.This component effectively functions as a rollup, which could be:</p><ol><li><p>A fully independent minimal (ZK) rollup</p></li><li><p>A general-purpose (EVM) rollup</p></li></ol><div class="relative header-and-anchor"><h3 id="h-a-completely-independent-zk-rollup"><strong>A completely independent (ZK) rollup</strong></h3></div><p>The first approach involves designing a dedicated minimal Keystore rollup (MKSR) that leverages ZK to batch Keystore signer updates and achieve <em>relatively fast finality*</em>. The entire Keystore state would be stored in an indexed/sparse Merkle Tree, enabling efficient commitment by publishing its root on L1. To ensure correctness when transitioning between commitments, the rollup would also submit a ZK proof (along with the new Keystore state commitment) which would be verified on L1. The different authorize logic would be implemented as ZK circuits and each Keystore identifier (user) would specify which ZK circuit (ECDSA keys, P256 keys, multiownable, multisig etc.) to execute to authorize their update.</p><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>relatively fast finality*</u>: Achieving near-instant (&lt;10s) finality is currently hindered by several technical limitations. Specifically, proof generation requires several minutes, and the Keystore state commitment stored on L1 becomes visible to L2s only after a delay of several minutes.</em></p></div></div></div></div><p>This design was described by Vitalik in <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://notes.ethereum.org/@vbuterin/minimal_keystore_rollup"><u>this article</u></a>. <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="mailto:michael.dehoog@coinbase.com"><u>Michael de Hoog</u></a>, Base Principal Software Engineer, presented a technical implementation in <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://hackmd.io/@mdehoog/mksr"><u>this post</u></a> relying on low level PLONK circuits and eventually, after iterating over it, we switched to <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/xenoliss/keyspace"><u>using ZK VMs</u></a>. The switch to ZK VMs was essential to simplify proof aggregation (through unbounded recursion), streamline circuit development (by enabling Rust coding instead of low-level Gnark), and leverage continuous performance and usability enhancements made by contributors in this field daily.<br><br>After experimenting with this approach here are the pros and cons we identified:</p><ul><li><p>Pros:</p><ol><li><p>We had full control over the MKSR, allowing us great flexibility to reach our goals.</p></li><li><p>We achieved relatively fast finality, making Keystore state changes visible to all chains after a few minutes.</p></li><li><p>This agnostic approach did not take part into specific teams or protocol, favoring a future enshrinement into the EVM.</p></li></ol></li><li><p>Cons:</p><ol><li><p>We had to deal with the technical challenges associated with implementing a ZK rollup from scratch.</p></li><li><p>We faced a tradeoff between achieving fast finality and driving the price per Keystore update down. To mitigate this we envisioned to make the rollup either using a proof aggregator or being an aggregator itself. The goal was to increase the amount of things being proven at once before submitting it to L1.</p></li><li><p>In order to mitigate the risk of a critical bug in a ZK VM, we had to implement multi-proving and were planning to support SP1, RiscZero and Intel TEE.</p></li><li><p>The wallet teams willing to use the Keystore needed to split their logic between Rust code (for the authorize logic implemented as a ZK circuit), and Solidity code embedded in their wallet contract (that would read the latest Keystore commitment known, extract the user’s signer configuration and ensure it was used to sign the UserOp being executed).</p></li></ol></li></ul><p><br>After testing this approach, we found that the relatively fast finality achieved through ZK didn’t justify the complexity trade-offs and technical limitations it introduced (such as rapid signature expiration). To encourage adoption by smart wallet teams, we needed near-instant finality (&lt;10s).</p><div class="relative header-and-anchor"><h3 id="h-a-general-purpose-evm-rollup"><strong>A general purpose (EVM) rollup</strong></h3></div><p>We concluded that the only way to make Keystore state updates immediately available across chains was to:</p><ol><li><p><strong>Set the Keystore state</strong> on relevant chains by replaying state update transactions.</p></li><li><p><strong>Use cross-chain syncing </strong>to synchronize the Keystore state when directly setting is not possible.</p></li></ol><p>From this observation, implementing the Keystore on a general-purpose EVM rollup seemed like a suitable solution. In fact, our second approach builds on insights from the first implementation:</p><ul><li><p>It simplifies Keyspace by utilizing an existing rollup to host the Keystore state and logic.</p></li><li><p>It allows setting the Keystore state to make updates immediately available on relevant chains.</p></li><li><p>It implements cross-chain syncing<strong> </strong>to synchronize the Keystore state when setting is not possible.</p></li></ul><p><br>In this design, the Keystore consists solely of smart contracts deployed across all supported chains. One chain is designated as the master chain, hosting the master Keystore, while the others function as replica chains, each hosting a replica Keystore.</p><p><br>We began implementing an early version of this approach <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/xenoliss/keyspace-v2"><u>here</u></a>. However, this version still had a major issue we faced in the MKSR solution: signatures expiring prematurely due to the master Keystore state being “compressed” into a single commitment (the master chain state root). After digging into commitments and cryptographic accumulators for a few days we came to the conclusion that there is no solution that allows all of the following at the same time:</p><ol><li><p>Commit to a (unbounded) set of elements.</p></li><li><p>Generate inclusion and exclusion proof.</p></li><li><p>Given a commitment C1, generate an inclusion proof P1 for an element E1, change the value of another element E2 and generate a new commitment C2, have P1 still verifying against C2.<br></p></li></ol><p>We addressed this issue by realizing that a globally unique Keystore was no longer necessary. For this reason, consistency (over the Keystore state across chains) was not mentioned in the pros and cons above and will instead be explored in the <u>Keystore consistency</u> section.</p><p>However, this new approach provides a better overall wallet user experience (for both users and developers). <strong>The remainder of this article will focus on the general-purpose EVM rollup approach</strong> and its unique set of challenges.</p><p></p><hr><div class="relative header-and-anchor"><h1 id="h-3-keystore-enabled-wallet">3. Keystore Enabled Wallet</h1></div><div class="relative header-and-anchor"><h2 id="h-integrating-keystore">Integrating Keystore</h2></div><p>As of now, integrating the Keystore into an existing wallet involves three steps:</p><ol><li><p>Select a master chain.</p></li><li><p>Inherit the Keystore contract.</p></li><li><p>Properly set state.</p></li></ol><div class="relative header-and-anchor"><h3 id="h-select-a-master-chain">Select a master chain</h3></div><p>As explained in above, the Keystore architecture relies on selecting a master chain as the source of truth. This master chain provides the canonical Keystore state, from which replica Keystores can synchronize their state from when needed.<br></p><p>While there are no specific restrictions on which chain can be chosen as the master one, a few key considerations should guide this decision:</p><ol><li><p>The security of the Keystore is tied to the security assumptions of the master chain.</p></li><li><p>The Keystore’s security also depends on the method used to synchronize state with replica chains. Whenever possible, prioritize “trustless” solutions over oracle-based ones to enhance reliability and security.</p></li></ol><div class="relative header-and-anchor"><h3 id="h-inherit-the-keystore-contract">Inherit the Keystore contract</h3></div><p>It is important to note that, in the current design, the wallet acts as its own Keystore. The next step, therefore, is to extend the wallet contracts by directly inheriting the Keystore contract.</p><p>While doing so, the wallet team will need to implement the following specific methods:</p><ol><li><p><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/Keystore.sol#L213"><u>_extractConfigHashFromMasterChain</u></a>: Implements the logic required to extract the signer configuration hash from the master chain, ensuring it can be accessed from any supported replica chain.</p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/Keystore.sol#L225"><u>_hookIsNewConfigAuthorized</u></a>: Implements the logic for authorizing new signer configurations. This corresponds to the authorize method described earlier.</p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/Keystore.sol#L242"><u>_hookApplyNewConfig</u></a>: Implements the logic to parse and cache the new signer configuration directly in the wallet contract’s storage. This caching improves UX by eliminating the need to provide a master chain storage proof with every wallet transaction or UserOp. Additionally, this hook can be used to execute arbitrary logic, such as performing wallet upgrades when appropriate.</p></li><li><p><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/Keystore.sol#L196"><u>hookIsNewConfigValid</u></a>: Implements the logic to validate that a new signer configuration is valid. This optional hook helps ensure that the wallet cannot be bricked by unintentionally updating to an invalid signer configuration.</p></li></ol><p>The current repository includes examples for:</p><ul><li><p>Implementing <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/chains/OPStackKeystore.sol#L93"><u>_extractConfigHashFromMasterChain</u></a> when the master chain is an OPStack chain.</p></li><li><p>Implementing a simple Keystore signer configuration logic for a <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/examples/op-stack/MultiOwnableWallet.sol"><u>MultiOwnableWallet</u></a>.</p></li></ul><div class="relative header-and-anchor"><h3 id="h-properly-set-state">Properly set state</h3></div><p>Above, we presented a high-level schema demonstrating how to maintain a good wallet UX—comparable to current standards—by allowing users to manually set their Keystore signer configuration to different chains.</p><p>In fact, setting state is similar to the replay-transaction mechanism implemented by the Coinbase Smart Wallet team, with one crucial distinction: instead of replaying a transaction, it replays the entire signer configuration (i.e., the Keystore state). This distinction highlights the primary purpose of the Keystore: directly synchronizing the state rather than the transaction history.</p><p>On the master chain, no special handling is required as the signer configuration will always reflect the latest state. Updates to the signer configuration originate from transactions sent to the master chain, where the source of truth is maintained.</p><p>On replica chains setting the state is left to the discretion of the wallet team. However, a few key considerations can be taken into account:</p><ol><li><p><strong>Revoking a signer</strong>: When revoking a signer, the updated state must be immediately set <em>to all replica chains*</em>. Failing to do so can significantly compromise wallet security, as a revoked signer could continue using the wallet, including upgrading it to a malicious implementation, on a replica chain.</p></li><li><p><strong>Prioritizing active chains</strong>: State update can be prioritized for chains that the user frequently interacts with, ensuring an optimal user experience on those chains.</p></li><li><p><strong>Lazily setting state:</strong> For replica chains that the user rarely interacts with, setting state can be deferred until just before they are used. This process can be bundled within a UserOp to minimize overhead.</p></li></ol><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>to all replica chains*</u>: we acknowledge that it may not be practical to replay revocations on all chains, both existing and future. In reality, the wallet provider may only replay revocations on the most relevant chain for the user, which could leave the wallet vulnerable to compromise on other chains where the signer revocation was not applied. The <u>Known issues and Limitations</u> section provides a more detailed explanation of the concept of “weak” signer revocation.</em></p></div></div></div></div><div class="relative header-and-anchor"><h2 id="h-unlocked-features">Unlocked features</h2></div><div class="relative header-and-anchor"><h3 id="h-signer-configuration-consistency">Signer configuration consistency</h3></div><p>By inheriting the Keystore contract, a wallet exposes all the necessary methods to instantly set its signer configuration or, when necessary, synchronize its state from the master chain.</p><p>Vitalik originally proposed <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://vitalik.eth.limo/general/2023/06/20/deeperdive.html#what-is-the-goal"><u>two methods</u></a> for implementing the Keystore:</p><ol><li><p><strong>Light version</strong>: Enforcing Keystore state consistency only during signer configuration updates.</p></li><li><p><strong>Heavy version</strong>: Enforcing Keystore state consistency for every transaction or UserOp.</p></li></ol><p>Both approaches aim to enforce state consistency at some point to provide a degree of security guarantees. However, the devil is in the details, and as we will cover in the <u>Keystore consistency</u> section, we believe that enforcing state consistency (regardless of the stage) introduces more drawbacks and fragility to the wallet than actual benefits.</p><p>For this reason, our approach intentionally avoids enforcing state consistency and encourages the following:</p><ol><li><p><strong>Manually set state</strong>: By setting the Keystore state, each wallet can cache its own keystore signer configuration. This ensures that, in most cases, users only need to send their original transactions without additional steps.</p></li><li><p><strong>Synchronize state as a fallback</strong>: The Keystore also provides the necessary cross-chain messaging primitives to synchronize the user’s signer configuration across chains if setting the state is not possible (i.e., it relies on state that is only available on the master chain).</p></li><li><p><strong>Bonus</strong>: On the master chain, the Keystore is essentially free, as it serves as the source of truth for the user’s signer configuration.</p></li></ol><div class="relative header-and-anchor"><h3 id="h-foundation-for-advanced-features">Foundation for advanced features</h3></div><p>By replicating the entire signer configuration state—instead of relying solely on transaction-based replay—integrating the Keystore into a wallet unlocks advanced features that were previously impossible or required significant effort to implement. Although this approach opens up various use cases, we expect its primary application to be in implementing Recovery Guardians.<br><br>It is well known that users can lose access to their keys. In such cases, the ability to trigger a recovery process (similar to those in Web2) is highly desirable. Features such as time-delayed recoveries or, more recently, <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://prove.email/"><u>ZK email</u></a> recoveries can significantly enhance the user experience during recovery scenarios.</p><p>Typically, recovery guardians rely on reading state from the blockchain to authorize recoveries—for example, verifying when a recovery was initiated or checking a DKIM certificate registry to validate ZK email recoveries. In a multi-chain wallet, this would normally require that the necessary data is available on every supported chain. However, with Keyspace, we replay the keystore’s latest full state rather than individual signer changes, allowing the recovery process to be executed on one chain and then bridged to others.</p><p></p><hr><div class="relative header-and-anchor"><h1 id="h-4-keystore-limitations">4. Keystore Limitations</h1></div><p>While integrating Keystore into wallets provides cross-chain synchronization of signer configurations and enables the development of new features, it also introduces its own set of challenges.</p><div class="relative header-and-anchor"><h2 id="h-maintaining-counterfactual-addresses">Maintaining counterfactual addresses</h2></div><p>The use of counterfactual addresses is familiar to anyone using EOAs. You can create a new keypair off-chain, safely share your public key, and start receiving funds at the corresponding address.&nbsp;</p><p>Wallets aim to replicate this functionality by leveraging <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://docs.soliditylang.org/en/v0.8.28/control-structures.html#salted-contract-creations-create2"><u>CREATE2 opcode</u></a> to generate counterfactual addresses. While CREATE2 allows users to know their wallet address in advance and receive funds on chains even if the wallet is not yet deployed, it also introduces significant drawbacks.</p><div class="relative header-and-anchor"><h3 id="h-factory-limitations">Factory limitations</h3></div><p>Since CREATE2 can only be invoked through smart contracts, it requires the use of a factory contract. To generate the same address, CREATE2 relies on the following:</p><ol><li><p><strong>Dependency on the Same Factory</strong>: The same factory contract must be used across all chains indefinitely. Once a factory is used to deploy wallets, it cannot be easily replaced with a new version for existing users. While making the factory upgradeable is an option—since only the factory contract address is relevant—this would require adding permissions to manage the upgrades, which is often undesirable. Moreover, as explained in the next point, upgrading the factory might not even be a practical solution for existing users.</p></li><li><p><strong>Fixed Contract Bytecode</strong>: The deployed contract’s bytecode (the initcode) must be unchanged. The initcode includes both the actual contract deployment code and the constructor parameters. As a result, once a user deploys their wallet to a specific chain, maintaining the same address on new chains requires deploying the exact same implementation <em>with the identical constructor parameters*</em>.</p></li></ol><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>with the identical constructor parameters*</u></em>: <em>this raises concerns about the risk of bricking the wallet when deploying to new chains, particularly if the user no longer has access to their original signer configuration—possibly due to updates made a long time ago. We expect state set transactions to be replayable (we strongly recommend ensuring replayability), which should significantly reduce the likelihood of such issues. If the state cannot be set, synchronization from the master Keystore should be available as a fallback to unbrick the wallet.</em></p></div></div></div></div><div class="relative header-and-anchor"><h3 id="h-chain-agnostic-keystore">Chain agnostic Keystore</h3></div><p>As discussed in the previous section, CREATE2 relies on deploying the same initcode to produce consistent addresses. While this imposes constraints on wallet constructor parameters—such as requiring wallets to always be deployed with the same initial signer configuration—it also introduces complexity into the Keystore implementation.</p><p>Since a wallet functions as its own Keystore (inheriting the Keystore contract), the Keystore contract code must remain chain-agnostic to function seamlessly across any chain it is deployed on. This requires addressing two key challenges:</p><ol><li><p><strong>Method Exposure on Different Chains</strong>: The Keystore must expose different methods depending on whether it is deployed on the master chain or on replica chains.</p></li><li><p><strong>Cross-chain State Reading</strong>: Regardless of which replica chain the wallet is deployed on, it must be able to read the signer configuration from the master chain so that synchronization can still be used as a fallback when setting state is not possible. The logic for cross-chain state reading heavily depends on the specific chain being read from (i.e., the replica chain). However, implementing separate logic for each replica chain is not feasible, as it would result in different wallet contract bytecodes and, consequently, different deployment addresses.</p></li></ol><p>While the first point is relatively straightforward to handle (e.g., using block.chainid to determine whether the wallet is deployed on the master chain and enabling or disabling relevant methods accordingly), the second one is more nuanced.</p><p>As discussed in the <u>Cross-chain State Reading</u> appendix, the ability of replica chains to read from the master chain is closely tied to their ability to access L1 state. However, different replica chains use varying methods to retrieve the L1 state (e.g., obtaining a recent L1 state root).&nbsp;</p><p>Our current approach implements various methods that different chains use to access a recent L1 state root. When synchronizing, users can provide a generic <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/libs/l1-state-root/L1StateRootLib.sol#L28"><u>L1StateRootProof</u></a> specifying the logic to execute for retrieving the recent L1 state root. From this L1 state root, the state of the Keystore on the master chain can be extracted.</p><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em>An example of this approach is the implementation of the </em><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base-org/keyspace/blob/v0.1.0/contracts/src/core/chains/OPStackKeystore.sol#L93">_extractConfigHashFromMasterChain</a><em> method. Internally, it invokes the </em><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/chains/OPStackKeystore.sol#L103"><u>L1StateRootLib.verify</u></a><em> method, which manages the generic retrieval of a recent L1 state root.</em></p><p><em>Currently, the “generic” logic supports extracting the state root from the </em><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://docs.google.com/document/d/1VCNYgAOUXQp790ZAAncRGSb2cIaOE2sXgpOMhLXhZ70/edit?tab=t.0#heading=h.ituj3hdtowef"><u>L1Block</u></a><em> contract on OPStack chains and from the </em>BeaconStateRoots<em> contract for chains that are </em><a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://docs.google.com/document/d/1VCNYgAOUXQp790ZAAncRGSb2cIaOE2sXgpOMhLXhZ70/edit?tab=t.0#heading=h.oc8ao1uw9zb9"><em><u>EIP-4788</u></em></a><em> compatible.</em></p><p><em>Additional mechanisms for retrieving recent L1 state roots can be implemented.</em></p></div></div></div></div><div class="relative header-and-anchor"><h2 id="h-keystore-consistency">Keystore consistency</h2></div><p>While leveraging Keystore wallets provides a simple way to manage signer configurations across chains, one might initially think that Keystore also ensures consistency of wallet signer configurations across all chains. The primary motivation for this assumption would be to enable effective signer revocation across all chains. Although this was indeed a goal of our initial implementation, we realized that it introduces more drawbacks than benefits.</p><p>Ideally, we aim for the following:</p><ol><li><p>The master Keystore signer configuration to be immediately visible on all replica chains.</p></li><li><p>Keystore signer configuration consistency to be enforced at all times –the primary motivation for this would be to have effective signer revocation across all chains.</p></li></ol><p>The first point depends on the finality of the master chain. In the current implementation, consistency could only be enforced “eventually,” with the minimum relevant interval being the time it takes for the master chain to settle on L1.<br></p><p>The second point is more complex. Enforcing eventual consistency introduces significant wallet bricking risks. As discussed in the <u>Cross-chain State Reading</u> appendix, to read the state from the master chain, replica Keystores must implement logic to:</p><ol><li><p><strong>Access the L1 state root</strong>: this could be achieved via the <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://docs.google.com/document/d/1VCNYgAOUXQp790ZAAncRGSb2cIaOE2sXgpOMhLXhZ70/edit?tab=t.0#heading=h.ituj3hdtowef"><u>L1Block contract on OPStack</u></a>, by <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://docs.google.com/document/d/1VCNYgAOUXQp790ZAAncRGSb2cIaOE2sXgpOMhLXhZ70/edit?tab=t.0#heading=h.oc8ao1uw9zb9"><u>leveraging EIP-4788</u></a>, or through equivalent solutions.</p></li><li><p><strong>Verify L1 storage proofs</strong>: the Keystore must support verifying L1 storage proofs to confirm the state of the master chain.</p></li><li><p><strong>Verify the master chain storage proofs</strong>: the Keystore must support verifying master chain storage proofs to confirm the state of the master Keystore.<br></p></li></ol><p>However, either of these steps could fail due to:</p><ol><li><p><strong>Hard forks on the replica chains</strong>: a hard fork could deprecate the L1Block contract or equivalent solutions, leaving the Keystore unable to access a recent L1 state root.</p></li><li><p><strong>L1 storage changes</strong>: a switch to Verkle Trees for L1 storage could break Keystore implementations reliant on verifying L1 MPT proofs.</p></li><li><p><strong>Hard forks on the master chain</strong>: a hard fork on the master chain could alter how it settles its state on L1, rendering all replica Keystores unable to verify the master Keystore storage proof.</p></li></ol><p>These scenarios are likely as chains require regular upgrades to evolve and improve. If such changes occur, wallets deployed to replica chains might become stuck. Users would be unable to prove their master Keystore signer configuration without upgrading their wallet implementation to support the new proving logic. However, upgrading is permissioned and guarded by a consistency check, creating a chicken-and-egg problem.</p><p>A reliable way to recover from such a scenario would be to use native withdrawal and deposit workflows to propagate the Keystore signer configuration from the master chain to the replica chains. However, this approach requires chain-specific (or ecosystem-specific, such as for the OPStack) implementations, which cannot be easily modified to support new chains after the wallet has been deployed. An existing wallet deployed to a new chain might be unable to synchronize its Keystore configuration from the master chain if the deposit flow was not implemented in the initial version.<br></p><p>For these reasons, we currently believe that enforcing consistency—whether immediate or eventual, and whether for all actions or only for signer configuration updates—is neither worth the complexity it introduces nor the risk of bricking users' wallets.</p><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em>We acknowledge that our current solution is imperfect, as it does not address the signer revocation issue. However, we believe the ecosystem is not yet ready to effectively solve this problem, as stronger cross-chain messaging primitives that are guaranteed to be hardfork-resistant are still needed.&nbsp;</em></p><p><em>To clarify, Keyspace currently </em><strong><em>does not provide signer revocation guarantees across chains</em></strong><em>. Instead, it offers utility methods that allow users to easily propagate their current Keystore signer configuration across chains, which wallet teams can leverage to implement advanced features.</em></p></div></div></div></div><p><br></p><div class="relative header-and-anchor"><h1 id="h-5-keyspace-present-and-future">5. Keyspace Present and Future</h1></div><div class="relative header-and-anchor"><h2 id="h-present">Present</h2></div><p>Our <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/tree/v0.1.0"><u>implementation</u></a> of the <u>Keystore</u> offers nice improvements over existing wallet solutions. However, it still has known issues and limitations that we aim to address in future versions of Keyspace, as cross-chain state reading capabilities continue to evolve.</p><div class="relative header-and-anchor"><h3 id="h-key-benefits">Key benefits</h3></div><p>The current implementation of the Keystore:</p><ul><li><p>is based on a <u>“master-replica” chain architecture</u>.</p></li><li><p>consists of a <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://github.com/base/keyspace/blob/v0.1.0/contracts/src/core/Keystore.sol"><u>single smart contract</u></a> that wallets can inherit.</p></li><li><p>allows cross-chain state sharing of wallet signer configurations.</p></li></ul><p>This implementation introduces two key advancements over existing wallet solutions:</p><ol><li><p><strong>Simplified configuration consistency</strong>: implementing the Keystore exposes the logic required to maintain the user’s signer configuration consistent across chains.</p></li><li><p><strong>Support for generic state synchronization</strong>: our example implementation demonstrates how the implementation address of the user’s wallet can be stored in their Keystore configuration. Similarly, wallet vendor teams can leverage the Keystore to develop additional features by incorporating custom data into the configuration.</p></li></ol><div class="relative header-and-anchor"><h3 id="h-known-issues-and-limitations">Known issues and limitations</h3></div><p>When designing the architecture of the Keystore, we had to balance two key considerations:</p><ol><li><p><strong>Resistance to chain upgrades</strong>: ensuring wallets will not get bricked due to future chain upgrades.</p></li><li><p><strong>Effective cross-chain signer revocation</strong>: enabling consistent and reliable signer revocation across all chains (including future ones).</p></li></ol><p>The latter requires enforcing eventual consistency for all actions on the user’s wallet. However, as discussed in the <u>Keystore consistency</u> section, enforcing this comes at a cost, as it poses a non-negligible risk of bricking the user’s wallet in the event of chain upgrades. Currently, the available cross-chain state readin<u>g</u> solutions make the two considerations outlined above <em><u>mutually exclusive</u>*</em>—unless a trusted party is introduced to perform permissionless upgrades of all the user’s wallets.</p><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>mutually exclusive</u>*: this limitation can be partially mitigated by relying on native deposits and withdrawals to send cross-chain messages. However, as explained in the <u>Keystore consistency</u> section, this approach does not scale effectively across different L2 stacks.</em></p></div></div></div></div><p></p><p>Since chain upgrades are inevitable and there is no clear timeline for significant improvements in cross-chain state reading, <strong>we designed the initial implementation of the Keystore to prioritize resilience to chain upgrades, albeit at the expense of <em><u>weak signer revocation*</u></em>.&nbsp;</strong></p><p><br>While this is a genuine issue, it is important to highlight that the exact same issue also affects traditional EOAs (which lack any concept of revocation) as well as current wallet implementations (which currently do not allow revoking signers). Future versions of Keyspace will focus on addressing this issue to provide enhanced cross-chain consistency and strong signer revocations.</p><div data-type="callout" type="info"><link rel="preload" as="image" href="https://paragraph.xyz/editor/callout/information-icon.png"><div class="callout-base callout-info" data-node-view-wrapper="" style="white-space:normal"><img src="https://paragraph.xyz/editor/callout/information-icon.png" class="callout-button"><div class="callout-content"><div><p><em><u>weak signer revocation*</u></em>: "weak" in this context means that revoking a signer does not automatically apply to all existing and future chains. Instead, the revoked signer must be "manually" revoked on all relevant chains.</p></div></div></div></div><div class="relative header-and-anchor"><h2 id="h-future">Future</h2></div><p>The long-term goals of Keyspace are to:</p><ol><li><p>Establish a unique source of truth for wallet configurations.</p></li><li><p>Provide strong guarantees for signer revocation.</p></li><li><p>Enable near-instant finality, ensuring that wallet configuration changes are immediately reflected across all chains.</p></li><li><p>Remain cost-efficient and affordable to use.</p></li></ol><p>The path to achieving these goals relies on three major milestones.</p><div class="relative header-and-anchor"><h3 id="h-efficient-and-resilient-l1-state-reading">Efficient and resilient L1 state reading</h3></div><p>As detailed in the <u>Cross-chain State Reading</u> appendix, the ability to share state across chains relies heavily on each chain's capacity to access L1 state. Currently, the ecosystem is fragmented, with different chains supporting various methods for accessing L1 state (e.g., via an <u>L1 block hash</u> or <u>leveraging EIP-4788</u>). Some chains, like Arbitrum at the time of writing, lack support for trustless L1 state access entirely (unless using deposit transactions from L1 which are expensive).<br></p><p>Moreover, all these methods depend on constructing proofs (typically MPT proofs) offchain and submitting them onchain for verification. These proofs are likely to break in the future as L1 evolves its state representation (e.g., transitioning to Verkle Trees).<br></p><p>Addressing these challenges represents a critical first step in improving Keyspace. Achieving this will require the implementation and widespread adoption of new proposals such as L1SLOAD, the more ambitious REMOTESTATICCALL, or equivalent solutions that are resistant to L1 hard forks.</p><div class="relative header-and-anchor"><h3 id="h-moving-to-a-dedicated-keystore-rollup">Moving to a dedicated Keystore rollup</h3></div><p>Once the ability to read L1 state from any layer is resolved, the next challenge is reading the state of the master chain from its commitment posted to L1. Similar to reading L1 state, we aim to make this process as robust as possible against potential master chain upgrades that could affect how their state is committed to L1.<br></p><p>Allowing wallet vendor teams to select their own master chains facilitates quick adoption of Keyspace (aligned with a "ship fast" spirit), but it is not an ideal solution. Different master chains use various methods to commit their state to L1 and will evolve and upgrade independently, making the Keyspace ecosystem increasingly complex to manage. As these chains improve and upgrade, the methods for proving their state from their commitments posted to L1 will also diverge, further complicating the system.</p><p></p><p>The second major step would be to (re)introduce a Minimal Keystore Rollup to serve as the universal master chain for all wallet vendor teams. With this implementation, Keyspace’s resilience would depend solely on upgrades to this single chain (for example, the method of committing its state to L1). This approach simplifies the architecture by transitioning from n replica chains relying on m master chains to n replica chains relying on a single universal master chain, effectively reducing the complexity from an n² problem to an n problem.&nbsp;<br></p><p>There has been considerable hype recently around both based and native rollups—both of which could drastically benefit the MKSR and ease its implementation:</p><ol><li><p>Making the MKSR a <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://ethresear.ch/t/based-rollups-superpowers-from-l1-sequencing/15016"><u>based rollup</u></a> would simplify its sequencing by inheriting L1’s liveness and decentralisation.</p></li><li><p>Making the MKSR a <a target="_blank" rel="noopener noreferrer nofollow ugc" class="dont-break-out" href="https://ethresear.ch/t/native-rollups-superpowers-from-l1-execution/21517"><u>native rollup</u></a> would enable efficient verification of Keystore state transitions for batches of updates.<br></p></li></ol><p>Switching to a universal master chain is also essential for achieving the final milestone, transforming Keyspace into the comprehensive solution we envision.</p><div class="relative header-and-anchor"><h3 id="h-enshrine-keyspace-into-the-evm">Enshrine Keyspace into the EVM</h3></div><p>Finally, once Keyspace relies on a single universal Minimal Keystore Rollup (MKSR), we can envision it being enshrined into the EVM in the future. In this scenario, all nodes would run an MKSR node alongside an L1 node, providing seamless access to the MKSR via an L1SLOAD-equivalent opcode: KEYSTORESLOAD.</p><p>This final milestone would usher in a new era of native, user-friendly, multichain wallets by completely eliminating the need to provide MKSR state proofs.</p><p></p><hr><div class="relative header-and-anchor"><h1 id="h-6-conclusion">6. Conclusion</h1></div><p>The Keystore, as we envision it in the long term, is an important innovation for enhancing wallet user experiences in an increasingly multichain world. By unifying the management of wallet signers across different chains, it not only streamlines interactions but also improves security through signer revocation—a feature that traditional EOAs simply don’t offer today.</p><p>However, for the Keystore to reach its full potential—including enabling signer revocation, which is crucial for mitigating risks when keys are compromised and adds a much-needed layer of protection—a critical challenge must be addressed: the lack of a standardized, robust, and accessible method for retrieving L1 state from L2 networks. Without this cross-chain accessibility, implementing a consistent and secure solution remains out of reach. Moving forward, overcoming this obstacle is essential for realizing the benefits of account abstraction and ensuring that users enjoy a seamless, secure, and unified wallet experience.<br></p><p>As we continue to explore and innovate, collaboration across the ecosystem will be key to bridging these technological gaps. Ultimately, addressing these issues will pave the way for a more resilient and user-friendly multichain future.</p><p></p><hr><div class="relative header-and-anchor"><h1 id="h-appendix-cross-chain-state-reading">Appendix: Cross-chain State Reading</h1></div><p>When presenting the <u>Keystore architecture</u>, we assumed that L2s can, or will eventually be able to, access L1 state. This typically involves retrieving a recent commitment to verify a provided L1 state root. Once the L1 state root is verified, any storage slot can be validated by generating a corresponding storage proof.</p><p></p><p>Currently, most L2s can retrieve recent L1 block hashes, allowing them to prove recent L1 state (by verifying a full L1 block header).
